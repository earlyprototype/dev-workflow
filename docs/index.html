dev-workflow

A method for building software with an AI collaborator: two independent contracts, and the runnable tooling that enforces them.

issue-first fresh-context review verdict-gated merge self-watching PRs fails closed

Overview

Every change is governed by two contracts. The requirements contract fixes what must be true and traces it to tests. The delivery contract carries the change to main through gates enforced by tooling rather than convention. A unit of work is defined in the first and moves through the second.

A unit of work: scoped in the requirements contract, delivered through the delivery contract.

The two contracts

They are separated because they answer different questions and scale differently. A small change needs little of the first and a light pass of the second; a change to money, personal data, or anything irreversible needs the full weight of both.

Requirements contract

A single source of truth, a scoped and traceable module PRD, a completeness checklist that renders a dropped requirement as a visible unticked box, acceptance criteria that become tests (forward traceability is mandatory and machine-checked), and a build loop that validates the plan before code.

Delivery contract

Issue → branch → PR → gated review → verdict-gated merge → the agent receiving its own PR's outcome. Each stage that can be enforced by tooling is, and each gate fails closed: an error or a missing proof resolves to blocked, never allowed.

Delivery pipeline

The six stages every change passes through. The gates apply to every change; the effort inside them scales to the risk the change carries.

flowchart LR S1["1 · Issue"] --> S2["2 · ADR log"] --> S3["3 · Branch"] --> S4["4 · Gated review"] --> S5["5 · Verdict merge"] --> S6["6 · Watch outcome"]

1 Issue first

The approach is recorded before work starts and closed with the outcome. The number threads through branch, commits, and PR.

2 ADR log

Each decision is recorded as taken — context, decision, consequences, and the condition that would reopen it.

3 Branch

One branch per issue, one writer per branch. main is protected with an empty bypass list.

4 Gated review

A fresh-context review before done, and an automated review on the PR. CRITICAL / HIGH / MEDIUM blocks; LOW does not.

5 Verdict merge

One classifier reads the review verdict and arms merge only on a clean one — mechanically, not by comment.

6 Watch outcome

The agent watches its own PR and is re-invoked with the result, in the context that produced the change.

Verdict-gated merge

The automated review ends with AUTOMERGE_VERDICT: PASS or BLOCK. One classifier is the single definition of a clean review, used both to fail the required review check on a blocking verdict and to arm GitHub's auto-merge only on a genuine PASS. A finding therefore blocks the merge as a failed status check, not as a comment a human must notice.

flowchart TD Rev["Automated review"] --> V{"AUTOMERGE_VERDICT"} V -- "PASS" --> Clean["Classifier: clean"] --> Arm["Arm auto-merge"] --> Merge[("Merge once CI and conversations green")] V -- "BLOCK" --> NotClean["Classifier: not clean"] --> Fail["review check fails"] --> Hold["Branch protection holds the merge"] Skip["Reviewer self-skip (edits review machinery)"] --> NotClean Skip -. "check stays green" .-> Manual["Owner may merge by hand"]
The classifier is fetched from the base branch, so a PR cannot edit the code that judges it. A skip is never auto-merged, but stays manually mergeable.

Reviewed-outcome feedback

A PR's review lands minutes after the agent's turn ends, on GitHub, with no connection back to the machine. The agent starts a bounded, read-only poll of its own PR; because the harness re-invokes a session when its background process exits, the review's completion re-invokes the agent with a single classified outcome.

sequenceDiagram participant A as Agent participant G as GitHub participant H as Harness A->>G: Open PR A->>A: Background the watch (poll, bounded) G->>G: Run CI + automated review G-->>A: Review lands — poll process exits H->>A: Re-invoke with the outcome Note over A: act on the result

The classified outcome is one of:

merged → advance the board blocked → fix findings (printed inline) ci-failed → fix the build needs-human → a self-skipped review

A PostToolUse hook triggers the watch after gh pr create; the watch cannot run from the hook itself, because a hook-spawned process has no tracked task for the harness to re-invoke.

Requirements build loop

Before code, and through it. The requirements sign-off ships as its own PR, separate from and preceding the implementation PR, so no build effort is spent before a requirement is approved.

flowchart LR O["Observe"] --> P["Propose"] --> R["Adversarial review"] --> Al["Align"] --> B["Build, test-first"] B -. "next unit of work" .-> O

Observe

Read the scoped contextualisation manifest; produce a written summary of what was read and the constraints that follow.

Propose

A coverage map (criterion → test), a deviations list, and the single visible behaviour the unit will demonstrate.

Adversarial review

A second reader checks that nothing in the contract is uncovered and that no decision is presented as mere craft.

Align

A human resolves what was flagged: a gap changes the plan; an ambiguity returns to the source of truth.

Build

Test-first, against the completed contract — one visible behaviour, built end-to-end and shown working.

The toolkit

The runnable wiring, at the paths it occupies in a target repository — copy .claude/ and .github/ across unchanged. Each guard fails closed and ships its own tests, run in CI.
